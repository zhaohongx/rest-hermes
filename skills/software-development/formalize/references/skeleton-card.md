## 一、S6 防幻觉加固

### 1.1 必标 ⚠️ 的 6 类场景

| # | 类别 | 判定 | 示例 |
|---|------|------|------|
| 1 | 未验证的库/API/版本 | 使用了未在用户需求中明确指定的第三方依赖 | "基于 React 19 实现"——若用户未指定版本 |
| 2 | 物理/数学简化假设 | 对现实世界做了理想化抽象 | "假设无摩擦""假设均匀分布" |
| 3 | 数据规模不明的经验值 | 给出了具体数值但无来源 | "建议 batch_size=32"——若用户未提供数据规模 |
| 4 | 跨平台/版本兼容性 | 涉及多平台或多版本的行为 | "在 Windows/Linux/Mac 上行为一致" |
| 5 | 性能/SLA 承诺 | 给出了可量化的性能指标 | "响应时间 <100ms"——若未经验证 |
| 6 | 安全相关默认值 | 涉及安全的配置项 | "默认允许跨域""使用 SHA1 哈希" |

### 1.2 禁用词黑名单

以下词汇**不可出现在规格书主体**（⚠️ 描述段除外）：

```
大概 应该 可能 差不多 或许 似乎 一般来说 通常 基本上 现代 主流 流行
```

### 1.3 状态符号约束

规格书中维度状态**必须**使用 `✓` / `⚠️` / `✗`，不可替代为其他符号（如 ✅/❌/❗）。

## 二、S7 自检清单（完整版）

生成规格书后，逐项核对：

```
[ ] S0 跳过判定执行了吗？
[ ] 模式选对了吗（skip/ask/spec）？
[ ] 复杂度判对了吗（L1/L2/L3）？
[ ] 一级章节齐了吗？
[ ] 状态符号用 ✓⚠️✗ 而非其他吗？
[ ] 置信度格式是 **置信度**: 高/中/低 吗？
[ ] 有禁用词吗（且不在 ⚠️ 上下文中）？
[ ] STEM 题含 B1-STEM 段吗？
[ ] 失败模式库的强制要求满足了吗？
[ ] 隐含假设都标 ⚠️ 或 ✓ 了吗？
[ ] 追问数符合复杂度对应上限吗？
[ ] 矛盾约束有没有先于形式化被识别？
[ ] 命中失败模式 6/7/8 时，强制要求都满足了吗？
[ ] 登录类是否枚举了 >=3 种登录方式？
[ ] 导入类是否给了失败处理 + 进度可观测？
```

### 自检报告格式（必须附加在输出末尾）

```
---
[自检] ✓ 章节齐全 ✓ 置信度合规 ✓ 无禁用词 ✓ 模式匹配 OK
```

### 不合规处理

1. 在心中重写一次
2. 输出修正版
3. 自检报告标明哪一项被修复

**硬约束**：每次输出最多重写 1 次，避免死循环。

## 三、S7.1 反例触发机制（v3.1 新增）

> 防自检套话化的关键机制。

```
若 ⚠️ 数量 = 0 且 复杂度 ≥ L2：
  → 这是 FALSE NEGATIVE 信号
  → 没有一个真实的复杂规格能做到零不确定性

强制重审以下来源：
1. 隐含假设：是否接受了用户未声明的条件？
2. 外部依赖：是否假定了某个库/API/版本的行为？
3. 性能/SLA：是否假定了规模而未经证实？
4. 边界条件：空输入/超大输入/并发/超时真的都覆盖了吗？

重审后必须新增至少 1 个 ⚠️（除非输入确实穷尽了一切细节）
```

## 四、S8 置信度评估量表

| 等级 | 判定条件 | 必须输出 |
|------|---------|---------|
| **高** | 10 维度中 ≥8 ✓，无关键 ⚠️ | 置信度声明 |
| **中** | 10 维度中 ≥6 ✓，1-3 个关键 ⚠️ | 置信度声明 + 关键 ⚠️ 列表 |
| **低** | <6 ✓ 或 ≥4 关键 ⚠️ | 置信度声明 + "降低不确定性的下一步" |

低置信度时，"降低不确定性的下一步"必须是**可执行的具体动作**（如"送样做 XRD 鉴定硅赋存形态"），不可为泛泛的"进一步研究"。

## 五、输出长度约束表

| 类型 | 目标长度 | 超限判定 |
|------|---------|---------|
| 跳过 | < 100 字 | >200 字 → 违规 |
| 追问 | < 400 字 | >600 字 → 违规 |
| L1 微型 | 300-600 字 | — |
| L2 标准 | 800-1800 字 | — |
| L2 + STEM | 1200-2500 字 | — |
| L3 扩展 | 1500-3500 字 | — |

**反模式**：简单问题写 5000 字 / 复杂问题写 500 字 / 章节缺失。

## 六、追问数收敛表

| 输入复杂度 | 追问数硬上限 |
|-----------|-------------|
| 简单（单实体单需求） | ≤ 1 个 |
| 中等（2-5 实体） | ≤ 2 个 |
| 复杂（>5 实体或系统级） | ≤ 3 个 |

**合并规则**：若初步识别出上限数以上问题，必须检查能否合并。禁止通过拆分单一关注点凑数。

## 七、失败模式对照表

| 失败模式 | 表现 | 修正 |
|---------|------|------|
| 该跳过没跳 | 给原子修改输出完整规格 | 强化 S0 |
| 该追问没追 | 对"做个智能系统"直接形式化 | 强化 S1 |
| 追问过多 | >3 个问题 | 收敛到 ≤3 |
| 简单问题过度 | L1 用 L3 模板 | S2 分级 |
| 模板飘忽 | 章节标题随意改 | S7 自检 |
| STEM 缺公式 | 物理题没列方程 | B1-STEM |
| 假装确认 | 未验证 API 没标 ⚠️ | S6 |
| 模糊词泛滥 | 充斥"大概/应该" | S7 自检 |
| 置信度缺失 | 末尾无置信度 | S7 自检 |
| 推荐缺冷启动 | 推荐系统未提 | 失败模式 1 |
