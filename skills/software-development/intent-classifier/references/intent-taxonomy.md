# intent-classifier 意图分类法（17 类详解）

> **定位**：定义每类意图的触发特征、边界 case、常见混淆。供分类器 LLM 兜底时参考。

---

## Group A: DIRECT_RESPOND

### A1: chitchat

**定义**：寒暄、问候、无实质信息交换。
**触发词**：你好、hi、hello、在吗、谢谢、再见、晚安
**反例**："你好，帮我查一下..." → 这不是 chitchat，后续有实质性请求
**混淆风险**：confirm — 确认类有明确的前置上下文

### A2: confirmation

**定义**：对前一条消息的确认、复述、或推进信号。
**触发词**：好的、OK、继续、嗯、对、确认、行、可以
**触发条件**：长度 ≤10 字 且 命中确认词典
**混淆风险**：chitchat — "好的"可能是对别的事的礼貌回应

### A3: emotional_support

**定义**：情绪表达、倾诉，不寻求技术方案。
**触发特征**：情绪词（开心、难受、焦虑、郁闷、生气）+ 第一人称
**示例**："我今天心情不好""工作压力好大"
**反例**："这个报错让我很焦虑，帮我看看" → 本质是 debug

### A4: simple_qa

**定义**：事实类问答，答案确定、不需要设计或推理。
**触发特征**：疑问词 + 常识/百科类实体
**示例**："巴黎是哪个国家的""Python 是什么意思""今天周几"
**反例**："如何设计一个推荐系统" → system_design

### A5: translation

**定义**：显式要求语言转换。
**触发词**：翻译、translate、用英文说、翻成、译成
**示例**："把这段翻译成英文""translate to Japanese"
**反例**：内含翻译需求但主要是别的意图

### A6: summarization

**定义**：显式要求压缩/概括内容。
**触发词**：总结、概括、摘要、归纳、提炼
**示例**："总结一下这篇文章""概括一下要点"

### A7: rewrite

**定义**：改写、润色、调整语气/风格。
**触发词**：改写、润色、改得、调成、换成...风格
**示例**："把这段改得正式点""润色一下这段话"

### A8: creative_writing

**定义**：创作文本、诗歌、故事等，不涉及工程任务。
**触发词**：写诗、编故事、创作、写篇、写首
**示例**："写一首关于秋天的诗""编个科幻故事"
**关键判断**：输出是创意文本，不是代码/规格/方案

## Group B: TOOL_EXECUTE

### B1: shell_command

**定义**：显式 CLI 命令。
**触发特征**：命令名开头 + 参数/选项
**示例**：`pytest tests/ -v` / `kubectl apply -f deploy.yaml` / `git log`

### B2: atomic_edit

**定义**：具体、精确的代码修改（动作+位置+目标三要素齐全）。
**触发特征**：
- 动作：改、修改、替换、删除、加、重命名
- 位置：文件名（含扩展名）/行号/函数名
- 目标：明确的新值/结果
**示例**："把 user.py 第 42 行的 print 改成 logger.info"
**反例**："优化一下 user.py" → 目标不明确 → ambiguous_requirement

### B3: file_operation

**定义**：文件系统操作。
**触发词**：新建文件、创建目录、删除文件、移动文件、重命名文件
**示例**："新建文件 config.py""删除 tmp 目录"

## Group C: SKILL_INVOKE

### C1: code_design

**定义**：设计或规划代码结构、架构。
**触发特征**：设计 + 代码/模块/组件/接口
**示例**："设计一个缓存层""规划一下模块结构"
**路由**：code-design / plan skill

### C2: code_review

**定义**：审查代码质量、安全性、规范性。
**触发词**：审查、review、检查这段代码、看看有什么问题
**示例**："review 这个 PR""检查这段代码的安全性"

### C3: debug

**定义**：排查错误、分析异常。
**触发特征**：报错/异常/崩溃/不工作/为什么 + 错误描述
**示例**："为什么运行时报 500 错误""这个函数不工作了"

### C4: system_design

**定义**：设计系统架构、方案。
**触发特征**：设计 + 系统/架构/方案/平台 + 多模块/多服务
**示例**："设计一个消息队列""规划微服务架构"
**与 formalize 关系**：通常需要 formalize 前置（L3 复杂度）

---

## Group D: FORMALIZE_REQUIRED

### D1: ambiguous_requirement

**定义**：需求描述模糊，核心对象/目标/约束缺失。
**触发条件**：
- 核心对象缺失（"优化一下"——优化什么？）
- 目标不明（"做个智能系统"——智能到什么程度？）
- 关键约束缺失（"做个排序"——数据规模？实时性？）

### D2: contradictory_constraints

**定义**：需求中包含矛盾约束（词对匹配或语义张力命中）。
**矛盾词对**：免费×企业级 / 简单×功能全面 / 毫秒级×AI / 灵活×零配置
**语义张力**：功能数≥5但声称"简单" / 时间<1周但子系统≥3

### D3: high_complexity

**定义**：需求复杂度高，直接动手风险大。
**触发条件**（任一）：
- 实体（名词短语）>5 个
- 系统级关键词命中（服务/集群/分布式/微服务/多端）
- 涉及 2+ 独立子系统

### D4: cross_domain

**定义**：需求跨越多个专业领域。
**触发条件**：同时涉及 2+ 领域（如"前端+后端+运维"、"算法+工程"）
**示例**："做一个 ML 模型训练的 Web 管理平台"→ ML + Web + DevOps
