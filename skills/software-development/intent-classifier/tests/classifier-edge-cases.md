# intent-classifier 边界踩坑档案

> **定位**：收录"曾经判错过的场景"，防止同类误判复发。每一条来源于真实踩坑或代码评审发现。
> **维护规则**：每发现一次 misclassification，在此文件追加一条用例，标注日期和根因。

## 误判档案

### E001: 创意写作被误判为 system_design

**日期**：2026-05
**输入**："写一首关于分布式系统的诗"
**误判**：`system_design`（关键词"分布式系统"命中）
**正确**：`creative_writing`（"写一首...诗"是明确的创作意图）
**根因**：系统关键词优先级过高，未先检查创作信号
**修复**：在 Layer 3 关键词扫描中，`creative_writing` 优先于 `system_design`

### E002: 知识问答被误判为 debug

**日期**：2026-05
**输入**："什么是内存泄漏？为什么会导致程序崩溃？"
**误判**：`debug`（"崩溃"命中）
**正确**：`simple_qa`（反问句本质是知识问答）
**根因**：错误关键词"崩溃"未区分"描述故障现象"与"排查具体错误"
**修复**：`debug` 触发需附加条件——有具体代码/系统上下文

### E003: 确认信息被误判为 ambiguous

**日期**：2026-05
**输入**："OK 我明白了"
**误判**：`ambiguous_requirement`
**正确**：`confirmation`（≤10 字确认）
**根因**：Layer 1 长度判断在 `OK` 后未匹配完整确认短语
**修复**：在确认词典中补充"OK"和"OK 我明白了"的前缀匹配

### E004: 业务需求与代码设计混淆

**日期**：2026-05
**输入**："设计一个积分系统"
**误判**：`system_design`
**正确**：`code_design` 或 `ambiguous_requirement`（积分系统可能是简单功能，非系统级）
**根因**：`设计`+"系统"直接跳 D 组，但"积分系统"实体数少、非分布式
**修复**：`system_design` 增加复杂度门控——需同时命中 ≥2 个系统级信号

### E005: Shell 命令中混杂自然语言

**日期**：2026-05
**输入**："帮我跑一下 pytest tests/ -v --cov"
**误判**：`ambiguous_requirement`（"帮我跑一下"）
**正确**：`shell_command`（命令主体明确）
**根因**：自然语言前缀干扰了正则匹配
**修复**：Layer 2 的正则匹配应容忍前导自然语言（如"帮我跑一下"、"请执行"）

### E006: 多意图混合导致 confidence 空转

**日期**：2026-05
**输入**："审查这段代码的安全性，然后写个测试覆盖异常路径"
**误判**：`code_review`（只取了 primary）
**正确**：`code_review`(primary) + `code_design`/`atomic_edit`(secondary)
**根因**：多意图识别逻辑未充分测试"审查+实现"组合
**修复**：secondary_intents 应在本次输入中出现"然后"、"接着"、"再"时强制扫描

## 运行说明

每条用例的验证：
1. 将 `输入` 传入 intent-classifier
2. 确认 `primary_intent` 不等于 `误判` 值
3. 确认 `primary_intent` 为 `正确` 值
4. 如果是多意图，确认 `secondary_intents` 包含正确的次要标签
